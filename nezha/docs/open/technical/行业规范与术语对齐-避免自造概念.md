# 行业规范与术语对齐 — 避免自造概念

**用途**：列出可与紫微对齐的**既有行业规范与术语**，对外与对标准时优先使用这些表述，避免创造新概念。  
**版本**：v1.0 | **日期**：2026-02-15

---

## 一、身份与标识（优先使用规范术语）

| 来源 | 规范术语 | 解释 | 紫微用法建议 |
|------|----------|------|--------------|
| **W3C** | **DID**（Decentralized Identifier） | 去中心化标识符，W3C 推荐标准（2022）；不依赖中心化注册机构，可验证、可解析为 DID Document。 | ✅ **已对齐**。文档与对外一律使用「DID」「DID 文档」，不说「智能体唯一 ID」等自造名。 |
| **W3C** | **DID Document** | DID 解析后得到的数据，含 verification method、service 等。 | ✅ 技术方案与接口中统一用「DID 文档」。 |
| **W3C** | **DID Method** | 某一类 DID 的创建、解析、更新、停用规则（如 did:ziwei:...）。 | ✅ 对外可说「紫微实现 did:ziwei 这一 DID Method」。 |
| **W3C** | **Verifiable Data Registry** | 存储 DID 及 DID 文档的可信数据存储（如分布式账本、数据库）。 | 可选：介绍链时可以说「私有链作为紫微的 Verifiable Data Registry」。 |
| **W3C** | **DID Resolution / DID Resolver** | 根据 DID 解析出 DID 文档的过程与组件。 | 可选：獬豸链上 DID 查询接口即「DID Resolution」能力。 |

**避免自造**：不说「AgentID」「智能体链上身份」等与 DID 重复的概念；统一用 **DID**。

## 二、访问控制与策略（优先使用规范术语）

| 来源 | 规范术语 | 解释 | 紫微用法建议 |
|------|----------|------|--------------|
| **NIST / ISO** | **ABAC**（Attribute-Based Access Control） | 基于属性的访问控制；决策依赖主体、资源、动作、环境等属性。 | ✅ **已对齐**。獬豸策略模型即 ABAC，对外与文档写「基于 ABAC」或「ABAC 策略」。 |
| **NIST / 通用** | **RBAC**（Role-Based Access Control） | 基于角色的访问控制。 | 若策略中有「角色」维度，可说「支持 RBAC 与 ABAC 结合」；不单独造「智能体角色模型」等词。 |
| **OASIS** | **XACML**（eXtensible Access Control Markup Language） | 策略描述与决策的 XML 标准。 | 紫微用 Cedar 而非 XACML；对外可说「采用 ABAC 策略模型，引擎实现为 Cedar」，不声称兼容 XACML。 |
| **通用** | **Policy / Authorization Policy** | 授权策略。 | ✅ 统一用「策略」「授权策略」，不说「智能体规则引擎」等模糊词。 |
| **通用** | **Principal, Resource, Action, Context** | ABAC 中常见的主体、资源、动作、上下文。 | ✅ 与 Cedar 模型一致；文档中可直接用这四类术语。 |

**避免自造**：不说「智能体权限引擎」「3AF 策略」等无法与行业对应的话；说 **ABAC / 策略引擎 / Cedar**。

## 三、数据与安全态势（沿用成熟品类名）

| 来源 | 规范/品类术语 | 解释 | 紫微用法建议 |
|------|----------------|------|--------------|
| **Gartner / 行业** | **DSPM**（Data Security Posture Management） | 数据安全态势管理：发现、分类、评估、保护数据资产。 | 紫微**不做**数据级 DSPM；对外说「我们做的是**智能体**安全态势管理，可类比为面向智能体的 ASPM」。 |
| **Gartner / 行业** | **DLP**（Data Loss Prevention） | 数据泄露防护。 | 紫微不做内容 DLP；可说「在智能体场景提供**行为级防护**（策略拦截、审批），与 DLP 理念类似但对象是智能体行为」。 |
| **Gartner / 行业** | **CIEM**（Cloud Infrastructure Entitlement Management） | 云基础设施权限管理。 | 紫微不做云 IAM 扫描；可说「智能体**身份与权限管理**，与 CIEM 理念类似，对象是智能体身份（DID）与权限」。 |
| **Gartner / 行业** | **UEBA**（User and Entity Behavior Analytics） | 用户与实体行为分析。 | 紫微不做「人」的 UEBA；可说「**智能体行为分析与审计**（Trace + 存证），与 UEBA 理念类似，对象是智能体」。 |
| **Gartner / 行业** | **CSPM**（Cloud Security Posture Management） | 云安全态势管理。 | 紫微**不涉及**；不声称做 CSPM。 |

**对齐原则**：对外用 **ASPM / ABP / AIEM / ABAA** 时，明确说明「对标/类比 DSPM、DLP、CIEM、UEBA，但对象是智能体」；不造 AGPM、AITM 等无出处的缩写。

## 四、审计与存证（沿用通用与合规术语）

| 来源 | 规范/通用术语 | 解释 | 紫微用法建议 |
|------|----------------|------|--------------|
| **ISO 27001 / 通用** | **Audit Trail / 审计迹** | 可追溯的操作记录序列。 | ✅ 统一用「审计迹」「操作审计」；Trace 可称「全行为审计迹」。 |
| **通用 / 法律** | **Non-repudiation / 不可抵赖** | 行为可证明、责任可认定。 | ✅ 链上存证即用于「不可抵赖」；对外可说「司法级不可抵赖」「可验真审计迹」。 |
| **通用** | **Merkle Tree / Merkle Root** | 哈希树与树根，用于批量数据完整性验证。 | ✅ 技术文档用「Merkle 树」「Merkle 根上链」；不说「存证树」等自造名。 |
| **通用** | **Immutable Log / 不可篡改日志** | 仅追加、不可修改或删除的日志。 | ✅ 链上存证即「不可篡改的审计迹」。 |
| **等保 / 国内** | **等保**（网络安全等级保护） | 国内等级保护制度。 | ✅ 规划中合规写「等保三级」即可；不造「智能体等保」等词。 |

**避免自造**：不说「操作指纹链」「3AF 存证」等；说 **审计迹、Merkle 存证、链上存证、不可抵赖**。

## 五、认证与联邦身份（标准协议术语）

| 来源 | 规范术语 | 解释 | 紫微用法建议 |
|------|----------|------|--------------|
| **IETF** | **OAuth 2.0 / OIDC**（OpenID Connect） | 授权与身份认证协议。 | 企业 IdP 对接时用「OIDC/OAuth2 联邦」；文档写「支持与企业 IdP（OIDC/OAuth2）联邦」。 |
| **IETF** | **SAML** | 企业 SSO 的断言与协议。 | 若支持 SAML，直接写「SAML 联邦」；不造「智能体 SSO」等。 |
| **IETF** | **SCIM**（System for Cross-domain Identity Management） | 跨域身份供应与同步。 | 若做组织/用户同步，写「SCIM 同步」；不造「智能体目录同步」等。 |
| **通用** | **Federation / 联邦** | 多域身份与信任联合。 | ✅ Matrix 联邦、身份联邦均用「联邦」；与行业一致。 |

**避免自造**：企业对接用 **OIDC、OAuth2、SAML、SCIM、联邦**，不造新协议名。

## 六、通信与互操作（标准与通用术语）

| 来源 | 规范/通用术语 | 解释 | 紫微用法建议 |
|------|----------------|------|--------------|
| **Matrix 规范** | **Matrix Protocol / Matrix 协议** | 开放联邦通信协议（端到端加密、多服务器）。 | ✅ 文档与对外统一「Matrix 协议」「Matrix 联邦」；不说「紫微通信协议」。 |
| **通用** | **E2EE**（End-to-End Encryption） | 端到端加密。 | ✅ 写「Matrix 端到端加密」或「E2EE」。 |
| **业界趋势** | **A2A**（Agent-to-Agent） | 智能体间通信与互操作。 | 可说「紫微通过 Matrix 与太白协议实现 A2A 互操作」；不造「紫微 A2A」等品牌化协议名。 |
| **通用** | **API / REST / Webhook** | 接口与回调方式。 | 技术文档直接用 REST、Webhook 等标准词。 |

**避免自造**：通信层说 **Matrix、E2EE、A2A、REST**；太白协议可称「基于 Matrix 的治理扩展」而非新发明一套「紫微协议」。

## 七、合规与认证（国内外标准名称）

| 来源 | 规范术语 | 解释 | 紫微用法建议 |
|------|----------|------|--------------|
| **ISO** | **ISO 27001** | 信息安全管理体系。 | 规划中写「ISO 27001」即可；不造「智能体安全体系」等。 |
| **国内** | **等保三级** | 网络安全等级保护三级。 | ✅ 文档写「等保三级」；不造「智能体等保」。 |
| **国内** | **商密**（商用密码） | 商用密码应用与认证。 | 规划中写「商密认证」即可。 |
| **通用** | **SOC 2** | 服务组织控制报告（美）。 | 若做国际客户可写「SOC 2」；不造新合规品牌。 |

**避免自造**：合规表述用 **ISO 27001、等保、商密、SOC 2** 等既有名称。

## 八、区块链与存证（通用术语）

| 来源 | 规范/通用术语 | 解释 | 紫微用法建议 |
|------|----------------|------|--------------|
| **通用** | **Permissioned Ledger / 许可链** | 需许可方可参与记账的链。 | ✅ 用「私有链」「联盟链」或「许可链」；不说「紫微链」。 |
| **通用** | **Consortium Blockchain / 联盟链** | 多机构共管的许可链。 | 多机构场景写「联盟链（如 Hyperledger Fabric、FISCO BCOS）」；不造链品牌名。 |
| **通用** | **Hash / Cryptographic Hash** | 密码学哈希。 | 技术文档用「哈希」「SHA3-256」等；不说「存证码」等。 |

**避免自造**：链相关用 **私有链、联盟链、许可链、哈希、Merkle**；DID 存储说「Verifiable Data Registry」或「链上 DID 注册」即可。

## 九、紫微可用的「组合表述」（仍基于规范术语）

对外一句话时，用**规范术语组合**，不造新概念：

- 「紫微基于 **W3C DID** 为智能体提供身份，采用 **ABAC** 策略与 **Matrix** 联邦通信，审计迹 **Merkle 存证** 满足**不可抵赖**与**等保**要求。」
- 「智能体**身份与权限管理**（对齐 CIEM 理念）、**行为防护与审批**（对齐 DLP 理念）、**行为分析与审计**（对齐 UEBA 理念），但对象是智能体而非数据/用户；身份采用 **DID**，存证采用**链上 Merkle**。」

---

## 十、不建议使用的自造或模糊表述

| 避免说 | 建议改为 |
|--------|----------|
| 智能体唯一 ID / AgentID（与 DID 重复） | **DID** |
| 智能体链上身份 | **DID** 或 **链上 DID 注册** |
| 3AF 策略 / 智能体规则引擎 | **ABAC 策略** / **策略引擎（Cedar）** |
| 操作指纹链 / 存证树 | **Merkle 树 / Merkle 根上链** / **链上存证** |
| 紫微通信协议 | **Matrix 协议** + **太白协议（基于 Matrix 的治理扩展）** |
| 智能体等保 / 智能体安全体系 | **等保三级** / **ISO 27001**（并说明适用于智能体治理场景） |
| AGPM、AITM 等无出处缩写 | **ASPM/ABP/AIEM/ABAA**（并说明对标 DSPM/DLP/CIEM/UEBA） |

---

**总结**：身份用 **W3C DID**，策略用 **ABAC / Cedar**，审计用 **审计迹 / Merkle 存证 / 不可抵赖**，通信用 **Matrix / E2EE / A2A**，合规用 **ISO 27001 / 等保 / 商密**，安全品类用 **DSPM/DLP/CIEM/UEBA** 类比并明确「对象是智能体」。这样与行业规范保持一致，避免创造新概念。
